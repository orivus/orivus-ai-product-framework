Why this standard
exists

What becomes scarce when code generation becomes cheap.

1. The situation changed

Artificial Intelligence now executes a large and growing share of engineering work. It writes code, tests, and documentation at machine speed. This is genuinely useful — and it quietly moves the risk.

When output was expensive, the scarce asset was the ability to produce code. Now that output is cheap, the scarce asset is the ability to keep a product coherent as it grows.

2. Why prompt-to-code fails at scale

Problem → Prompt → Code works for a single change. Repeated hundreds of times, without a governing source of truth, it produces architectural drift, duplicated behavior, and knowledge that lives in chat logs rather than in the product. The code compiles; the product decays.

3. The hypothesis

When AI writes most of the software, the most important asset stops being the code and becomes the Product Knowledge: the purpose, boundaries, architecture, contracts, and verified decisions that let a product keep evolving. 

Product Knowledge        →  permanent asset
Artificial Intelligence  →  current executor
Source code              →  derived artifact

When the executor changes, the knowledge remains. When the model improves, the architecture remains. When the session ends, the decisions remain.

4. What the standard proposes

  • Governance. Humans govern the product; AI governs engineering execution; Product Knowledge governs the code.
  • Governed execution. Work advances through Intentions and verified milestones — not unstructured prompts.
  • Evidence. Completion is proven by evidence, not claimed by confidence.
  • Evidence-based evolution. The standard itself only changes when real implementations show it must.

5. What it does not promise

It is not a prompt library, a coding agent, or an IDE plugin. It does not replace architecture, testing, or domain design. It does not depend on any specific product, vendor, or AI model.

6. Why now, and why experimental

This is v0.1 — deliberately frozen. It is published not because it is finished, but because it has reached the point where the best way to improve it is evidence from real use. It evolves through implementation and observable results, not through opinion.
